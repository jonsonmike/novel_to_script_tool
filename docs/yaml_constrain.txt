# AI 小说转剧本 — YAML Schema 设计文档
# 版本: v1.2.0
# 作者: 后端团队
# 日期: 2025-06-05

================================================================================
  一、为什么选择 YAML 作为剧本输出格式
================================================================================

1.1 可读性优先
  剧本的最终读者是人——作者、编剧、导演、演员。YAML 基于缩进的层级结构
  天然接近剧本排版习惯，作者可以直接在文本编辑器中阅读和修改，无需任何
  工具辅助。相比之下，JSON 的括号和引号在长文本中噪声极大。

1.2 结构化与自由文本的平衡
  纯文本（.txt）无法被程序解析，前端编辑器无法渲染结构化界面；JSON 过
  于机械，不适合人类直接编辑。YAML 恰好处于中间点：既有明确的层级结构
  供程序消费，又保留了足够的人类可编辑性。

1.3 行业趋势
  YAML 已成为影视行业数据交换的常见选择（如 Final Draft 的 .fountain
  格式衍生工具、剧本管理工具 Trelby 等均支持 YAML 导出）。选择 YAML
  降低了未来与专业工具对接的迁移成本。

1.4 技术考量
  - 支持多行字符串（| 和 > 语法），适合存储长台词和场景描述
  - 原生支持注释（#），方便编剧在文本中直接标注
  - 主流编程语言均有成熟解析库（Python: PyYAML, JS: js-yaml）
  - 可无损转换为 JSON，不会丢失任何结构信息


================================================================================
  二、顶层结构设计：为什么是 meta / characters / scenes 三部分
================================================================================

2.1 信息分离原则
  剧本数据按"变化频率"和"引用关系"自然分成三类：

  meta     — 全剧只出现一次的元数据（标题、改编范围、生成时间）
  characters — 跨场景复用的角色库（一个角色在多场戏中出现）
  scenes   — 线性展开的剧本正文（按时间顺序逐场排列）

  如果将所有信息平铺在一个数组中，每个场景都要重复角色的完整信息，
  导致数据冗余、一致性难以维护。例如角色"苏晚"在 20 个场景中出现，
  她的性格特征只应存储一次。

2.2 引用而非重复
  characters 库为每个角色分配全局唯一 ID（如 CHAR_LIN_MO）。
  场景中通过 characters_present 引用 ID，对话中通过 speaker_id 引用 ID。
  这样做的好处：
  - 数据体积小，避免重复
  - 修改角色信息时只需改一处
  - 前端可以建立角色索引，实现"点击角色名跳转到所有出场场景"

2.3 additionalProperties: false 的意义
  顶层的 additionalProperties: false 强制锁定数据结构。AI 模型有时会
  "创造性"地生成不在 Schema 中的字段。如果没有这个限制，AI 可能输出
  一个含有 novel_summary、author_bio 等多余字段的对象，前端解析时不会
  报错但会浪费带宽，且可能在不同版本间引入微妙的兼容 bug。
  严格模式确保"不符合 Schema 的输出 = 错误"，在开发阶段就能发现。


================================================================================
  三、meta 设计详解
================================================================================

3.1 novel_title vs script_title 分离
  novel_title 是原著的不可变标识，script_title 允许作者重新命名。
  实际场景中，一部小说可能被改编为多个版本（舞台剧版、电视剧版、
  网络短剧版），每个版本需要独立标题。分离两个字段为这种扩展留了空间。

3.2 user_instructions 的定位
  这是"人机协作"的关键载体。它记录了用户在触发 AI 转换前输入的个性化
  指令，存储于剧本中意味着：
  - 同一部小说用不同指令重新生成时，可追溯差异来源
  - 前端可回显上次设置，方便作者迭代调整
  - tone / focus_characters / custom_prompt 三个字段覆盖了从结构化选
    择到自由文本的全部输入场景

3.3 tone 不用 enum 的原因（v1.2.0 改动）
  v1.1.0 中 tone 是硬编码的 4 项 enum（正剧/喜剧/悲剧/悬疑），但在
  实际使用中作者可能要求"谍战正剧""古装奇幻喜剧"等组合风格。硬编码
  enum 会阻碍创意表达。v1.2.0 改为自由字符串，在描述中给出推荐值，
  既保留了引导功能，又不限制作者。

3.4 schema_version 的前向兼容设计
  剧本文件可能在不同时间用不同版本的工具生成。如果没有版本标记，将
  来 Schema 升级后旧数据无法判断该用哪套规则校验。这个字段允许读取
  端（前端编辑器、导出工具）根据版本号选择对应的解析逻辑。

3.5 generated_at 的时间戳意义
  同一部小说可能多次生成（修改指令、调参后重新生成）。时间戳区分不同
  版本的生成结果，支持后续的版本对比功能。


================================================================================
  四、characters 设计详解
================================================================================

4.1 唯一 ID 格式：CHAR_[A-Z0-9_]+
  选用 CHAR_ 前缀而非纯数字的原因：
  - 在剧本文本中搜索 "CHAR_LIN_MO" 不会和台词中的"林墨"撞关键词
  - 前缀明确标识这是系统生成的角色 ID，而非原著角色名
  - 正则约束防止 AI 生成格式不一致的 ID（如混用大小写、加空格等）

4.2 role_type 的三级分类
  "主角 / 配角 / 龙套"三级足以覆盖绝大多数网文的角色体系。
  不做更细的划分（如"反派主角""第一配角"）的原因：
  - 越细的分类越依赖主观判断，AI 分类准确率下降
  - 三级分类前端渲染时已经足够区分（页面布局通常只需 2-3 种样式）

4.3 traits 设计为字符串数组而非枚举
  角色性格是开放的创作概念——"腹黑""圣母""沙雕""病娇"等标签来自网文
  社区话语体系，无法穷举为固定枚举。数组形式的自由标签最灵活。

4.4 aliases 的意义
  小说中同一个角色可能有多个称呼（本名、绰号、化名、尊称）。如
  "林墨"可能被称为"墨儿""林公子""林大人""那位黑衣剑客"。AI 需要在
  实体提取阶段识别这些别名指向同一角色，aliases 字段存储识别结果，
  后续生成对话时就能准确标注 speaker_id。

4.5 physical_description（v1.2.0 新增）
  场景造型需要外貌参考。这个字段非强制（提取不到可留空），但存在时
  能显著提升导演/美术团队的使用体验。


================================================================================
  五、scenes 设计详解
================================================================================

5.1 为什么场景是基本叙事单位
  剧本天然以"场（scene）"为最小组织单元，而非"章（chapter）"。
  一章小说可能包含多个场景（从书房换到花园 = 两场戏），一个场景
  也可能跨越多个章节的闪回片段。以 scene 为基本单位更贴近剧本行业
  惯例，也方便前端按场次渲染分页编辑。

5.2 scene_id vs scene_number 的双编号设计（v1.2.0 改动）
  scene_id（如 "S0001"）：生成时分配，终身不变，用于数据库关联和引用。
  scene_number（如 1, 2, 3）：播放顺序号，编辑时会变（插入新场次后重排）。

  如果只有一个编号字段，编辑时删掉第 3 场后，第 4 场是否要改名？
  改名 = 破坏所有对"第 4 场"的外部引用；不改 = 编号出现空缺。
  双字段解决了这个矛盾。

5.3 characters_present 的引用约束（v1.2.0 改动）
  v1.1.0 中 items 仅为 string，可填任意值。v1.2.0 加了 pattern 约束，
  要求必须是 CHAR_ 前缀的合法 ID。这防止了：
  - AI 输出角色名而非角色 ID（如填 "林墨" 而非 "CHAR_LIN_MO"）
  - 拼写错误（如 "CHAR_LIN_MU"）导致前端查不到角色信息

5.4 time 字段的枚举选择
  "清晨/上午/下午/傍晚/夜晚/深夜/黎明"七个时段足以覆盖绝大多数场景的
  时间锚定需求。不做更精确的时间（如"下午 3:15"）的原因：
  - 网文原著很少给出精确时间，AI 无法提取不存在的信息
  - 精确时间对排练和排期的实际帮助有限
  - 模糊时段反而给导演更大的调度自由度


================================================================================
  六、content 混合类型数组设计（核心设计决策）
================================================================================

6.1 为什么不用分字段结构
  一个直观但错误的方案是给每个场景分配：

    dialogues: [...]    # 对话单独存
    actions: [...]      # 动作单独存
    sounds: [...]       # 音效单独存

  这破坏了内容的时序性。"苏晚拔剑 → 林墨说话 → 剑光闪过"这三个事件
  如果散落在不同的数组中，阅读者无法按时间顺序理解剧情。剧本的根本属
  性是"按时间展开的序列"，混合类型数组最忠实地保留了这个属性。

6.2 六种内容块类型的设计逻辑

  action      — 舞台指示和动作描述，是所有剧本的基础
  dialogue    — 角色对白，必须绑定 speaker_id
  voiceover   — 旁白/内心独白（v1.2.0 新增），与 dialogue 的区别在于
                它不一定有"说话的人"，可能是第三人称叙述
  transition  — 转场标记（v1.2.0 新增），如 FADE IN / CUT TO，虽然
                可以通过 action 表达，但独立类型让前端可以渲染特殊样式，
                也让导出模块可以生成行业标准格式
  sound       — 音效提示，独立于 action 以方便音效团队筛选
  note        — 编剧内部注释，渲染时应有视觉区分，导出为正式剧本时可隐藏

  是否还需要"灯光"（lighting）类型？目前可以通过 action ("灯光渐暗")
  或 note 覆盖。如果后续实际使用中发现大量灯光指令需要独立处理，再扩展
  enum——这是向后兼容的（旧数据不受影响，新类型只增不减）。

6.3 speaker_id 的适用规则
  - dialogue：必填。没有说话人的对白是数据错误。
  - voiceover：可选。如果旁白来自特定角色（如主角内心独白），填 ID；
    如果是第三人称全知叙述，不填。
  - action / transition / sound / note：不填。

6.4 ai_confidence 放在 content 块级别（v1.2.0 改动）
  v1.1.0 的 ai_confidence 在 meta 层，全剧本只有一个值。但 AI 的置信
  度在不同内容块上差异巨大：提取著名场景的对话时置信度高，生成次要配
  角的台词时置信度较低。颗粒度越细，作者越能精准定位需要修改的地方。
  数值的语义约定：
    0.9+  — AI 高度确定，通常可直接采用
    0.7-0.9 — 建议作者审阅
    0.5-0.7 — 可能需要修改
    <0.5   — AI 不确定，作者几乎肯定需要重写


================================================================================
  七、有意未纳入的设计
================================================================================

7.1 不做"原著文本对照"
  content 块中没有 original_text 字段存储对应的原著片段。原因：
  - 版权风险：逐句对照可能构成对原著的复制，而非改编
  - 数据膨胀：原著文本可能比剧本长 3-5 倍
  - 若后续需要，可通过 chapter_origin 做章节级别追溯，而非句子级别

7.2 不做角色关系图谱
  characters 中没有 relationship 字段（如"林墨 爱慕 苏晚"）。
  关系提取是信息抽取中的难题，准确率不足时反而产生误导信息。
  作者通过阅读剧本自然能理解人物关系，不需要 AI 画蛇添足。

7.3 不做分数/评分/热度
  meta 中没有 quality_score 或 confidence_score 等评价剧本质量的指标。
  AI 不擅长评判文学质量，硬加评分只会误导作者。ai_confidence 只表达
  "输出是否符合预期格式"，而非"这段写得好不好"。


================================================================================
  八、Schema 版本演进策略
================================================================================

8.1 版本号规则
  - 主版本号升级（1.x → 2.x）：破坏性变更（删除字段、修改枚举值）
  - 次版本号升级（1.1 → 1.2）：向后兼容的增量（新增字段、放宽约束）
  - 通过 schema_version 字段，读取端可识别版本并做兼容处理

8.2 可预期的 v1.3 / v2.0 扩展方向
  - 支持多幕结构（act 层级，在 scenes 之上再加一层分组）
  - 支持并行场景（同一时间不同地点的交叉剪辑）
  - 道具清单（props 字段，自动提取场景中的物品）
  - 多语言台词（对白原文 + 翻译版本）


================================================================================
  附录：与前端渲染的映射关系
================================================================================

  Schema 结构               →  前端组件
  ─────────────────────────────────────────────
  meta / meta.user_instructions  → 项目信息面板 + 改编参数回显
  characters[]                    → 角色列表面板（可筛选、搜索）
  scenes[n].scene_id + scene_number → 场景导航（时间轴/缩略图列表）
  scenes[n].characters_present    → 每场戏的角色头像/标签
  scenes[n].content[type]         → 内容块渲染路由：
    action      → ActionBlock（灰色底、斜体）
    dialogue    → DialogueBlock（角色头像 + 气泡 + 情绪标签）
    voiceover   → VoiceoverBlock（半透明遮罩样式）
    transition  → TransitionBlock（居中、加粗、小型大写）
    sound       → SoundBlock（喇叭图标 + 蓝色文字）
    note        → NoteBlock（黄色便签样式、可折叠/可隐藏）
  content[n].ai_confidence        → 置信度指示条（绿/黄/红色，hover 显示数值）
